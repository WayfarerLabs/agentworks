# Nerf Tools -- High-Level Architecture

## Overview

A nerf package is a YAML manifest that defines a family of scoped tool wrappers. A standalone
Python CLI reads manifests and generates self-contained shell scripts -- one per tool -- and
rulesync skill files. Those scripts are placed in a directory on the agent's PATH. The AI coding
framework's permission model controls which scripts the agent can invoke.

There is no runtime dependency on the manifest, no compiled binary, no SUID, no shared library.
Generated scripts are fully self-contained and human-readable.

## Components

```text
nerf (Python CLI)               standalone build and skill generation tool
  nerf build                    generate scripts from manifests
  nerf skill                    generate rulesync skills from manifests

nerfctl-claude-* (shell scripts) Claude Code permission management tools
                                generated by nerf build from the built-in nerfctl manifest
                                ship as part of the default manifest set

agentworks CLI                  thin wrapper -- controls nerf install on VMs via config
```

## Manifest Format

Each package has a `manifest.yaml` that defines a family of related scoped tools. YAML is used
over TOML or JSON because inline comments and multiline strings (for skill intro blocks) matter
for a file that operators write by hand.

### Package metadata

```yaml
package:
  name: nerf-git
  description: Scoped git operations for agents
  skill_group: nerf-git
  # Appears as the header of the generated rulesync skill.
  # Use this to explain when to use these tools, what they protect against,
  # and any important caveats. Plain text or markdown.
  skill_intro: |
    Use these tools instead of git directly. They enforce safe defaults and
    prevent destructive operations such as force pushes or pushes to protected
    remotes. If you need an operation not listed here, ask your operator.
```

### Tool definitions

#### Simple tool (fixed command, no parameters)

```yaml
tools:
  nerf-git-push-origin:
    description: Push the current branch to the origin remote
    command: [git, push, origin, HEAD]
```

#### Flag parameter

```yaml
tools:
  nerf-git-push-remote:
    description: Push the current branch to a non-origin remote
    example: nerf-git-push-remote --remote upstream
    command: [git, push, "{remote}", HEAD]
    params:
      remote:
        flag: --remote
        description: Name of the remote to push to (not origin)
        required: true
        pattern: "^[a-z0-9_-]+$"
        deny: [origin]
```

#### Positional parameter

```yaml
tools:
  nerf-git-fetch:
    description: Fetch from a specific remote
    command: [git, fetch, "{remote}"]
    params:
      remote:
        positional: true
        description: Remote to fetch from
        required: true
        pattern: "^[a-z0-9_-]+$"
```

#### Environment variables

```yaml
tools:
  nerf-az-account-show:
    description: Show the current Azure account
    command: [az, account, show]
    env:
      AZURE_CONFIG_DIR: /home/agentworks/.azure
```

### Full field reference

#### Tool fields

| Field         | Type     | Description                                                    |
| ------------- | -------- | -------------------------------------------------------------- |
| `description` | string   | Human-readable. Used in `--help` output and skill generation.  |
| `command`     | string[] | The command to execute. Use `{param}` for substitution points. |
| `example`     | string   | Optional. A concrete invocation shown in skill output.         |
| `params`      | map      | Parameter definitions keyed by name. See below.                |
| `env`         | map      | Environment variables to set before execution.                 |

#### Parameter fields

| Field        | Type           | Description                                                          |
| ------------ | -------------- | -------------------------------------------------------------------- |
| `flag`       | string         | Named flag (e.g. `--remote`). Mutually exclusive with `positional`.  |
| `positional` | bool           | If true, passed as a positional arg in declaration order.            |
| `description`| string         | Human-readable. Used in `--help` output and skill generation.        |
| `required`   | bool           | Defaults to false.                                                   |
| `pattern`    | string (regex) | Value must match this pattern.                                       |
| `allow`      | string[]       | Explicit allow-list. Value must be in this list.                     |
| `deny`       | string[]       | Explicit deny-list. Value must not be in this list.                  |
| `default`    | string         | Default value when not required and not provided.                    |

`pattern`, `allow`, and `deny` can be combined. All that apply are checked. `allow` and `deny`
cannot both be set on the same parameter.

## The `nerf` Build CLI

### Overview

`nerf` is a standalone Python CLI that lives in the agentworks repo but is usable independently.
It reads one or more manifests, merges them, and generates shell scripts and/or rulesync skills.
It runs on the operator's workstation; generated artifacts (scripts, skills) are deployed to VMs
via rsync. There is no Python or pipx dependency on the VM.

### Built-in catalog

`nerf` ships a catalog of built-in manifest groups (e.g. `nerf-git`, `nerfctl`). By default all
built-in groups are included -- this is opt-out, not opt-in. The nerftools CLI is permissive by
default; the agentworks VM config controls whether nerf is installed at all (opt-in).

### Manifest merging

Users can provide additional manifests on top of the built-in catalog. Merging is last-wins by
tool name: if two manifests define `nerf-git-push-origin`, the last one on the command line wins.
This makes it easy to override a built-in tool definition with a customized version.

```text
nerf build [--outdir <dir>] [--no-default] [--keep-existing] [<manifest> ...]
nerf skill [--outdir <dir>] [--no-default] [--keep-existing] [<manifest> ...]
```

- `--outdir <dir>`: directory to write output (default: current directory)
- `--no-default`: skip the built-in catalog entirely
- `--keep-existing`: preserve files in outdir that this run would not generate; default behavior
  clears the outdir before writing so removed tools do not linger
- `<manifest> ...`: additional manifest files, merged after the catalog in order

### Build-time validation

`nerf build` validates each manifest before generating any output:

- All `{param}` references in `command` have a corresponding `params` entry
- No param declares both `flag` and `positional`
- `allow` and `deny` are not both set on the same param
- `default` is not set when `required = true`
- `pattern` is a valid Python regex

Build fails with a clear error message on the first violation. No partial output.

### Generated scripts

Each generated script is self-contained: all argument parsing, validation, help text, and error
formatting is inlined. There is no shared library and no sourcing. Scripts behave identically to
the underlying tool from the perspective of exit codes and signal handling -- the final instruction
is always `exec`, which replaces the shell process with the target command.

Example for `nerf-git-push-remote`:

```bash
#!/usr/bin/env bash
# nerf-git-push-remote -- Push the current branch to a non-origin remote
# Generated from nerf-git manifest. Do not edit directly.

set -euo pipefail

usage() {
  cat >&2 <<'EOF'
Usage: nerf-git-push-remote --remote <remote>

  --remote    Name of the remote to push to (not origin)

Push the current branch to a non-origin remote.

Example: nerf-git-push-remote --remote upstream
EOF
  exit 1
}

REMOTE=""

while [[ $# -gt 0 ]]; do
  case "$1" in
    --remote) REMOTE="$2"; shift 2 ;;
    -h|--help) usage ;;
    *) echo "error: unknown argument: $1" >&2; usage ;;
  esac
done

if [[ -z "$REMOTE" ]]; then
  echo "error: --remote is required" >&2; usage
fi

if ! [[ "$REMOTE" =~ ^[a-z0-9_-]+$ ]]; then
  echo "error: --remote must match ^[a-z0-9_-]+$" >&2; exit 1
fi

if [[ "$REMOTE" == "origin" ]]; then
  echo "error: --remote cannot be 'origin'" >&2; exit 1
fi

exec git push "$REMOTE" HEAD
```

### Generated scripts as artifacts

Generated scripts are built from source (the manifest) and deployed. They are human-readable and
auditable. If a script needs to change, update the manifest and regenerate -- do not patch the
script by hand.

## Claude Code Permission Management (`nerfctl-claude-*`)

### Concept

`nerfctl-claude-*` are themselves nerf tools: shell scripts generated from the built-in `nerfctl`
manifest, shipped as part of the default manifest set. The `nerfctl-` prefix signals that these are
control-plane tools for operators, not tools for agents to invoke.

They manage Claude Code's `settings.local.json` (default) or `settings.json` (with `--settings`).

### Discovery

The tools look for `.claude/` in the current working directory only. If `.claude/` is not present
and no explicit `--settings` path is given, the tool exits with an error. There is no directory
traversal.

### Commands

```text
nerfctl-claude-grant <tool> [--settings <path>]
nerfctl-claude-deny  <tool> [--settings <path>]
nerfctl-claude-reset <tool> [--settings <path>]
nerfctl-claude-list         [--settings <path>]
```

### Operations

| Command                         | Effect                                                                   |
| ------------------------------- | ------------------------------------------------------------------------ |
| `nerfctl-claude-grant <tool>`   | Add `Bash(<tool>)` to `permissions.allow`, remove from `deny` if present |
| `nerfctl-claude-deny <tool>`    | Add `Bash(<tool>)` to `permissions.deny`, remove from `allow` if present |
| `nerfctl-claude-reset <tool>`   | Remove `Bash(<tool>)` from both `allow` and `deny`                       |
| `nerfctl-claude-list`           | Print all `Bash(nerf-*)` and `Bash(nerfctl-*)` entries from both lists   |

### Settings file format

```json
{
  "permissions": {
    "allow": [
      "Bash(nerf-git-push-origin)",
      "Bash(nerf-git-push-remote --remote *)"
    ],
    "deny": []
  }
}
```

For tools with parameters, `grant` writes the pattern with a wildcard for each parameter position
so that any valid argument passes. The nerf tool itself enforces parameter validation at runtime.

### `--settings` flag

Targets the specified file directly. The file need not be in the current directory.

### JSON dependency

The `nerfctl-claude-*` scripts use `jq` for JSON manipulation. `jq` is a required dependency for
these tools. It is available by default on Agentworks VMs.

## Rulesync Skill Generation

### What the manifest provides

The manifest captures everything needed to generate a complete skill without additional input:

- `package.skill_intro`: header text explaining the package, when to use it, and any caveats
- `tool.description`: what each tool does
- `tool.example`: a concrete invocation example (optional)
- `param.description`: what each parameter means
- `param.pattern`, `param.deny`, `param.allow`: constraints surfaced in the skill so agents know
  what values are valid before invoking

### Generated skill structure

`nerf skill` generates one `<skill_group>/SKILL.md` per package (rulesync skill directory
format). The file has YAML frontmatter followed by the skill content:

```markdown
---
name: nerf-git
description: "Scoped git operations for agents"
targets: ["*"]
---

# nerf-git

<skill_intro content>

## nerf-git-push-origin

Push the current branch to the origin remote.

**Usage:** `nerf-git-push-origin`

No arguments.

---

## nerf-git-push-remote

Push the current branch to a non-origin remote.

**Usage:** `nerf-git-push-remote --remote <remote>`

**Arguments:**

- `--remote` (required): Name of the remote to push to. Cannot be `origin`.
  Must match `^[a-z0-9_-]+$`.

**Example:** `nerf-git-push-remote --remote upstream`
```

### Integration with workspace provisioning

When Agentworks provisions a workspace with rulesync configured, it includes the relevant skill
files in the workspace's rulesync source based on which nerf packages are installed on the VM.

## Agentworks CLI Integration

The agentworks CLI does not own the nerf build process -- it exposes VM-level configuration that
controls how `nerf build` is invoked when provisioning or updating a VM.

Nerf tool installation is opt-in: disabled by default, enabled via `vm.config.install_nerf_tools =
true`. When enabled, VM config specifies:

- Whether to include the default catalog (`skip_nerf_defaults`)
- Any additional user manifests to merge in (`nerf_addl_manifests`)
- Whether to preserve files not regenerated by this run (`nerf_keep_existing`, default false)

**Build and deploy model:** `nerf build` and `nerf skill` run on the operator's workstation (where
the agentworks CLI runs). Agentworks rsyncs the output directories to the VM. There is no Python
or pipx dependency on the VM -- only bash and the generated shell scripts.

Executables are installed to `/opt/agentworks/nerf/bin/` (added to the agent PATH). Skills are
installed to `/opt/agentworks/nerf/skills/` (rulesync sources from this path). Custom output
locations can be specified via `nerf_bin_dir` and `nerf_skills_dir` in VM config.

### PATH configuration

Nerf tools live under `/opt/...` and must be available to all users on the VM, including agent
users created after init. The user-level `~/.agentworks-path.sh` mechanism used for install
commands is not sufficient here.

Agentworks writes `/etc/profile.d/agentworks-nerf.sh` (via sudo) with:

```bash
export PATH="/opt/agentworks/nerf/bin:$PATH"
```

This is sourced automatically for all users in login shells. SSH sessions are login shells by
default, so agent tool invocations pick up the PATH correctly.

## Directory Layout

```text
<nerf repo or package>/
  manifests/
    nerf-git/
      manifest.yaml
    nerf-az/
      manifest.yaml
    nerfctl/
      manifest.yaml       (defines nerfctl-claude-* tools)

/opt/agentworks/nerf/               canonical install root on VMs
  bin/                              generated executables (on agent PATH)
    nerf-git-push-origin
    nerf-git-push-remote
    nerf-az-account-show
    nerfctl-claude-grant
    nerfctl-claude-deny
    nerfctl-claude-reset
    nerfctl-claude-list
  skills/                           generated skills (rulesync source)
    nerf-git/
      SKILL.md
    nerf-az/
      SKILL.md
```

## Relationship to the Old Nerfed Commands Design

The `2026-03-08-nerfed-commands` design is superseded by this one. See
[its lockfile](../2026-03-08-nerfed-commands/locked.md) for the rationale. OS-level SUID
enforcement may appear as a separate future mechanism but is not part of this design.

| Aspect              | Old design (nerfrun)            | This design                           |
| ------------------- | ------------------------------- | ------------------------------------- |
| Enforcement         | OS-level SUID                   | Application permission model          |
| Compiled binary     | Required (Go or Rust)           | None                                  |
| Generated artifacts | Symlinks to nerfrun             | Self-contained shell scripts          |
| Credential access   | Inherited via SUID              | Agent's own access (injection: future)|
| RBAC                | rbac.toml checked by nerfrun    | Framework settings.json               |
| Bigred              | Lockfile checked by nerfrun     | Not in v1                             |
| Discovery (wcid)    | Built into nerfrun              | Not in v1                             |
| Audit logging       | nerfrun logs every invocation   | Not in v1                             |
